# Database Migration Strategy

## Overview

Popote uses **Drizzle ORM** with **PostgreSQL** for database management. Migrations run automatically in both development (run mode) and production (publish mode).

## How It Works

### Automatic Migration Execution

Migrations run automatically on **server startup** via SvelteKit's `hooks.server.ts`:

1. **First HTTP request** to the SvelteKit server triggers the `handle` hook
2. Hook checks if migrations have already run (via `migrationsRun` flag)
3. If not, runs all pending migrations from `app/src/lib/server/db/migrations/`
4. Sets flag to prevent re-running on subsequent requests
5. **Fail-fast behavior**: Server won't start if migrations fail

This approach works identically in:

- ✅ **Run mode**: `aspire start` (local development)
- ✅ **Publish mode**: Docker deployment (production)

### Connection String Resolution

The migration runner automatically resolves connection strings in this order:

1. `ConnectionStrings__popotedb` (Aspire injected, both run & publish modes)
2. `DATABASE_URL` (fallback for manual deployments)

## Files

### Core Migration Files

- **`app/src/lib/server/db/migrate.ts`**: Migration runner (exported function + CLI)
- **`app/src/hooks.server.ts`**: Server startup hook that runs migrations
- **`app/src/lib/server/db/migrations/`**: SQL migration files (generated by Drizzle)

### Migration Workflow

```bash
# 1. Modify schema
vim app/src/lib/server/db/schema.ts

# 2. Generate migration SQL
cd app && pnpm db:generate

# 3. Review generated migration
cat src/lib/server/db/migrations/XXXX_*.sql

# 4. Start Aspire (migrations run automatically on first request)
aspire start

# Optional: Run migrations manually (CLI mode)
cd app && pnpm db:migrate
```

## Manual Migration Execution

If needed, you can run migrations manually:

```bash
cd app
pnpm db:migrate
```

This uses the same `migrate.ts` file but runs in CLI mode (standalone execution).

## Deployment (Publish Mode)

When deployed via `aspire publish`:

1. Aspire generates Docker Compose configuration
2. Container starts with `ConnectionStrings__popotedb` environment variable
3. SvelteKit server boots
4. First request triggers migration hook
5. Migrations apply to production Postgres

**No additional configuration required** — same code path as development.

## Benefits of This Approach

✅ **DRY**: Single migration runner for both CLI and automatic execution  
✅ **Fail-fast**: Server won't start if migrations fail (prevents broken deployments)  
✅ **Aspire-native**: Works seamlessly in both run and publish modes  
✅ **No external tools**: No need for init containers or separate migration services  
✅ **Idempotent**: Drizzle tracks applied migrations, won't re-run

## Troubleshooting

### Migrations not running?

Check SvelteKit server logs for:

```
🗄️  Running database migrations...
✅ Database migrations complete
```

If you see:

```
❌ Migration failed: ...
```

This indicates a migration SQL error or connection issue.

### Manual verification

Connect to Postgres and check migrations table:

```bash
aspire describe db  # Get connection details
# Then use psql or pgAdmin to connect

SELECT * FROM drizzle.__drizzle_migrations;
```

### Rollback strategy

Drizzle migrations are **forward-only**. To rollback:

1. Write a new migration that undoes changes
2. Generate via `pnpm db:generate`
3. Restart server to apply

## Alternative Approaches Considered

### ❌ Init Container Pattern

- **Pros**: Clean separation of concerns
- **Cons**: Requires Aspire init container support (not in Vite resources), added complexity

### ❌ Separate Migration Service

- **Pros**: Dedicated resource for migrations
- **Cons**: Orchestration complexity, race conditions with app startup

### ✅ **Server Startup Hook (CHOSEN)**

- **Pros**: Simple, works everywhere, fail-fast, no orchestration needed
- **Cons**: Slight delay on first request (negligible for dev, one-time for production)

## Decision Record

**Date**: 2026-04-11  
**Decided by**: Ash (DevOps)  
**Status**: ✅ Implemented

Run migrations in SvelteKit server startup hook (`hooks.server.ts`) for both run and publish modes.

**Rationale**:

- Simplest approach that works identically in dev and production
- No Aspire-specific orchestration needed
- Fail-fast prevents broken deployments
- Idempotent via Drizzle's migration tracking
